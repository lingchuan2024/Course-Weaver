# AI 课件知识重构系统：完整建设方案

## 0. 项目一句话定义

这个项目不是做一个普通的 PDF/PPT 总结器，而是做一个 **AI 课件知识重构系统**：

> 将老师的 PDF/PPT 课件解析成结构化知识，再按照学生最容易理解的认知路径，生成一份低认知负荷、可追溯、可校验、便于复习的自然讲义式笔记。

它的核心目标不是“把课件变短”，而是：

> 把老师用于课堂展示的课件，重构成学生可以自学、复习、查漏和做题的学习材料。

---

## 1. 为什么不能只做普通总结？

老师的课件通常不是完整教材，而是课堂讲解的提纲。它经常存在以下问题：

1. **内容跳跃**：课件常常直接给定义、公式、图、伪代码，但省略动机和中间推理。
2. **强依赖口头讲解**：老师课堂上讲过的解释，PDF 里可能只剩几个关键词。
3. **图表和公式信息密度高**：很多关键知识在图、表、箭头、公式、颜色高亮里，普通文本摘要容易漏掉。
4. **顺序不一定适合自学**：课件顺序是老师展示顺序，不一定是学生理解顺序。
5. **普通 AI 总结不可验证**：AI 写得很顺，但用户不知道它有没有漏内容，也不知道哪些话来自课件，哪些是模型补充。

因此，这个项目要解决的不是“总结”，而是：

```text
无损解析课件
→ 抽取知识单元
→ 建立知识关系
→ 设计学习路径
→ 生成自然讲义
→ 覆盖率校验
→ 支持原文追溯
```

---

## 2. 核心设计原则

### 2.1 内部强结构化，外部自然表达

这是整个项目最重要的原则：

> **CourseIR 要硬，最终笔记要软。**

也就是说，系统内部必须像数据库一样严谨：

- 每页有哪些 block；
- 每个 block 属于标题、正文、公式、图片、表格还是伪代码；
- 每个知识点来自哪些页；
- 每个公式、图、算法有没有进入笔记；
- 每段笔记有没有来源；
- 哪些内容被合并、忽略、放入附录或标记为不确定。

但是用户最终看到的笔记不能像填表，不能变成：

```text
### 动机
### 定义
### 例子
### 易错点
```

最终笔记应该像一个好助教重新讲了一遍课件，具有自然的讲解节奏：

```text
这一节真正想解决的问题是：为什么直接递归会非常慢？

以切钢条问题为例，最自然的做法是枚举第一刀切在哪里，然后递归求剩下部分的最优收益。这个思路本身没有错，问题在于递归过程中会反复遇到相同的子问题……
```

后台用结构保证完整性，前台用讲义保证可读性。

---

### 2.2 不是按页总结，而是按知识点重构

坏的做法：

```text
第 1 页讲了……
第 2 页讲了……
第 3 页讲了……
```

这只是逐页摘要，不是好笔记。

正确做法是：

```text
先逐页抽取
再跨页合并
最后按知识点生成
```

例如 BFS 可能分布在多页：

```text
p.10：BFS 的动机
p.11：BFS 伪代码
p.12：BFS 运行例子
p.13：BFS 正确性
p.14：BFS 复杂度
```

最终笔记不应该写成五页摘要，而应该合并成一个完整知识点：

```text
BFS
├── 它解决什么问题
├── 为什么用队列
├── 伪代码如何运行
├── 为什么能求无权图最短路径
├── 时间复杂度为什么是 O(V+E)
└── 常见误解
```

---

### 2.3 主笔记负责好读，附录负责完整，覆盖报告负责可信

为了同时满足“好懂”和“不遗漏”，不能把所有细节都塞进正文。

应该分成三层：

| 层级 | 作用 |
|---|---|
| 讲义式主笔记 | 面向第一次学习，顺着读能理解 |
| 结构化复习包 | 面向复习，包括概念表、公式表、算法表、易错点、自测题 |
| 覆盖报告 | 面向可信性，说明每页每个 block 的去向 |

这样可以避免两种极端：

1. 主文太短，虽然好读但漏内容；
2. 主文太长，虽然完整但读起来像资料堆积。

---

## 3. 产品最终应该长什么样？

用户上传一份 PDF/PPT 后，系统输出的不只是一篇笔记，而是一个学习包：

```text
output/
├── 00_overview.md              # 一页总览
├── 01_learning_path.md         # 学习路径
├── 02_lecture_notes.md         # 自然讲义式主笔记
├── 03_concepts.md              # 概念表
├── 04_formulas.md              # 公式表
├── 05_algorithms.md            # 算法/伪代码解释
├── 06_figures.md               # 图表解释
├── 07_common_mistakes.md       # 易错点
├── 08_self_test.md             # 自测题
├── 09_source_index.md          # 原课件页码索引
└── 10_coverage_report.md       # 覆盖报告
```

其中最重要的是：

1. **02_lecture_notes.md**：真正给学生读的自然讲义。
2. **10_coverage_report.md**：证明系统没有明显遗漏的覆盖报告。
3. **09_source_index.md**：让学生可以从笔记回到课件原页。

---

## 4. 整体系统架构

系统可以拆成七层：

```text
1. 文件上传层
   PDF / PPT / Markdown / 图片课件

2. 文档解析层
   页面、文字、公式、图、表、伪代码、bbox、reading order

3. CourseIR 层
   PageIR、Block、KnowledgeUnit、Relation、CoverageLedger

4. AI 理解层
   知识单元抽取、跨页合并、关系判断、图表解释

5. Note Director 层
   判断讲解策略、组织学习顺序、控制详略和表达方式

6. 笔记生成层
   讲义式主笔记、结构化复习包、覆盖报告

7. 交互层
   PDF 对照、来源跳转、人工修正、局部重生成
```

最小链路是：

```text
PDF/PPT
→ blocks
→ knowledge_units
→ CourseIR
→ lecture_notes.md
→ coverage_report.md
```

## 5. 核心中间表示：CourseIR

CourseIR 是整个系统的核心资产。

它不是最终笔记，而是从课件中抽取出来的结构化知识表示。

### 5.1 CourseIR 总体结构

```json
{
  "course_id": "algorithm_lecture_12",
  "document": {
    "title": "Dynamic Programming",
    "source_file": "lecture12.pdf",
    "num_pages": 56
  },
  "pages": [],
  "blocks": [],
  "knowledge_units": [],
  "relations": [],
  "note_plan": [],
  "coverage_ledger": []
}
```

---

### 5.2 PageIR

每页解析后要保存：

```json
{
  "page_id": "p012",
  "page_number": 12,
  "width": 1280,
  "height": 720,
  "page_image": "pages/p012.png",
  "blocks": [
    {
      "block_id": "p012_b001",
      "type": "title",
      "text": "Dynamic Programming",
      "bbox": [80, 40, 900, 110],
      "reading_order": 1
    },
    {
      "block_id": "p012_b002",
      "type": "formula",
      "text": "OPT[i] = max(...)",
      "latex": "OPT[i] = \\max(...)",
      "bbox": [120, 200, 800, 260],
      "reading_order": 2
    },
    {
      "block_id": "p012_b003",
      "type": "figure",
      "image_path": "crops/p012_b003.png",
      "bbox": [100, 300, 1000, 620],
      "reading_order": 3
    }
  ]
}
```

必须保留：

```text
page_number
block_id
bbox
block_type
raw_text
image_crop
reading_order
```

这些字段是后续溯源、覆盖检查、PDF 高亮跳转的基础。

---

### 5.3 KnowledgeUnit

知识单元是 CourseIR 中最重要的对象。

```json
{
  "unit_id": "U_023",
  "name": "最优子结构",
  "type": "concept",
  "source_blocks": ["p012_b002", "p013_b001"],
  "source_pages": [12, 13],
  "raw_evidence": [
    "An optimal solution contains optimal solutions to subproblems."
  ],
  "importance": "core",
  "difficulty": "medium",
  "status": "extracted"
}
```

知识单元类型建议先定义为：

| 类型 | 含义 |
|---|---|
| concept | 概念 |
| definition | 定义 |
| formula | 公式 |
| algorithm | 算法 / 伪代码 |
| theorem | 定理 |
| proof | 证明 |
| example | 例子 |
| figure | 图示 |
| table | 表格 |
| comparison | 对比 |
| warning | 易错点 |
| exercise | 练习 |
| summary | 结论 |

---

### 5.4 Relation

知识点之间需要建立关系。

```json
{
  "relation_id": "R_001",
  "from": "U_recursion",
  "to": "U_dynamic_programming",
  "type": "prerequisite_of",
  "confidence": 0.87,
  "evidence_blocks": ["p003_b002"],
  "reason": "动态规划通常从递归问题中的重复子问题改造而来。"
}
```

关系类型建议：

| 关系 | 含义 |
|---|---|
| prerequisite_of | A 是 B 的前置知识 |
| part_of | A 是 B 的组成部分 |
| used_in | A 被 B 使用 |
| example_of | A 是 B 的例子 |
| derives | A 推导出 B |
| proves | A 证明 B |
| contrasts_with | A 和 B 需要对比 |
| confused_with | A 和 B 容易混淆 |

这一步决定笔记能不能从“课件顺序”重排成“学习顺序”。

## 6. AI 多次调用流程

这个项目必须多次调用 AI，但每次调用只做一件事，并且尽量输出结构化结果。

总体流程：

```text
全局扫描
→ 页面抽取
→ 跨页合并
→ 知识关系构建
→ Note Director 规划
→ 按知识点生成笔记
→ 章节整合
→ 覆盖校验
→ 局部修补
```

---

### 6.1 第一次调用：全局扫描

输入：

```text
整份课件的标题、目录、页缩略图、每页标题
```

输出：

```json
{
  "lecture_topic": "Dynamic Programming",
  "sections": [
    {
      "title": "Why Dynamic Programming",
      "pages": [1, 2, 3]
    },
    {
      "title": "Rod Cutting",
      "pages": [4, 5, 6, 7]
    },
    {
      "title": "Memoization and Bottom-up DP",
      "pages": [8, 9, 10]
    }
  ],
  "likely_main_units": [
    "递归重复计算",
    "最优子结构",
    "重叠子问题",
    "状态转移方程",
    "自顶向下备忘录",
    "自底向上动态规划"
  ]
}
```

目的不是写笔记，而是建立全局地图。

---

### 6.2 第二类调用：逐页抽取知识单元

输入：

```text
单页 PageIR + page image + blocks
```

输出：

```json
{
  "page_id": "p008",
  "knowledge_units": [
    {
      "type": "example",
```

> 尽量不漏。宁可多抽，不要少抽。

输入：

```text

```

输出：

```json
[
  {
```

目的：

> 帮助系统生成更符合学习顺序的笔记。

### 6.5 第五类调用：Note Director

这是解决“笔记太死板”的关键。

Note Director 不直接写最终笔记，而是决定每个部分应该怎么讲。

输入：

```text
CourseIR + merged_units + relations
```

输出：

```json
{
  "writing_strategy": [
    {
      "section_title": "为什么需要动态规划？",
      "units": ["U_recursion_problem", "U_overlapping_subproblems"],
      "style": "narrative_explanation",
      "goal": "先用递归重复计算的问题建立动机",
      "detail_level": "medium"
    },
    {
      "section_title": "从切钢条问题理解状态转移",
      "units": ["U_rod_cutting", "U_recurrence"],
      "style": "worked_example",
      "goal": "用具体例子引出状态定义和转移方程",
      "detail_level": "high"
    },
    {
      "section_title": "两种实现方式：备忘录与自底向上",
      "units": ["U_memoization", "U_bottom_up"],
      "style": "comparison",
      "goal": "对比两种 DP 实现方式",
      "detail_level": "medium"
    }
  ]
}
```

Note Director 需要决定：

| 情况 | 写作策略 |
|---|---|
| 第一次出现核心概念 | 动机 + 直觉 + 例子 + 定义 |
| 重复出现概念 | 简短引用，不重复展开 |
| 抽象公式 | 先解释来源，再解释符号 |
| 长伪代码 | 先讲目标，再分段解释 |
| 复杂图 | 先讲图意，再拆元素 |
| 证明 | 先讲证明思路，再讲关键步骤 |
| 边角内容 | 放入附录或脚注 |
| 易混淆概念 | 用对比表 |
| 考试常考点 | 放入复习提醒 |

---

### 6.6 第六类调用：生成讲义式主笔记

输入：

```text
Note Director 的写作策略 + 对应知识单元 + 来源 blocks
```

输出：

```markdown
这一节真正想解决的问题是：为什么有些递归程序会非常慢？

以切钢条问题为例，假设一根长度为 n 的钢条可以切成不同长度，每种长度都有对应价格，目标是让总收益最大。最直接的想法是枚举第一刀切在哪里，然后递归求剩余部分的最优收益。

这个递归想法本身没有错，问题在于它会不断重复遇到相同的子问题。比如长度为 10 的问题会用到长度为 6 的最优解，长度为 9、8、7 的问题也可能反复用到长度为 6 的最优解。如果每次都重新计算，递归树会迅速膨胀。

动态规划的核心思想就是：既然这些子问题会重复出现，那就把每个子问题的答案保存下来。之后再遇到同一个子问题时，直接查表，而不是重新递归。
```

主笔记应该是自然讲义式，而不是机械模板式。

---

### 6.7 第七类调用：生成结构化复习包

同一个 CourseIR 还可以生成：

```text
概念表
公式表
算法表
图表解释
易错点
自测题
```

这些可以结构化一些，因为它们不是主阅读材料，而是复习材料。

---

### 6.8 第八类调用：覆盖率校验

输入：

```text
原始 blocks + CourseIR + 笔记
```

输出：

```json
{
  "coverage_summary": {
    "valid_blocks": 132,
    "covered": 118,
    "merged": 8,
    "appendix": 3,
    "uncertain": 2,
    "missing": 1
  },
  "items": [
    {
      "block_id": "p012_b004",
      "block_type": "formula",
      "status": "covered",
      "note_location": "02_lecture_notes.md#从切钢条问题理解状态转移",
      "comment": "公式已保留，并解释了每个符号。"
    }
  ]
}
```

目的：

> 不是让 AI 自称没有遗漏，而是建立可检查的覆盖账本。

---

### 6.9 第九类调用：局部修补

如果覆盖报告发现 missing 或 uncertain，不要整份重写，而是局部修补。

例如：

```text
p.18 的图没有解释
p.23 的复杂度推导没有展开
p.31 的例子没有进入笔记
```

系统只针对这些地方生成 patch：

```json
{
  "patch_target": "02_lecture_notes.md#复杂度分析",
  "source_blocks": ["p023_b002", "p023_b003"],
  "patch_content": "这里需要补充复杂度推导……"
}
```

然后插入或合并到现有笔记中。

## 7. 最终笔记的形态设计

### 7.1 主笔记：自然讲义式

主笔记应该像“助教讲课”，而不是像数据库导出。

它应该：

1. 先讲问题，再讲概念；
2. 先讲直觉，再讲形式定义；
3. 用例子带出公式；
4. 公式出现后解释符号；
5. 算法出现前先讲目标；
6. 证明出现前先讲证明思路；
7. 易错点自然嵌入，而不是强行堆列表；
8. 重要来源用页码轻量标注。

主笔记示例结构：

```markdown
# 动态规划：把重复递归变成查表

这节课的核心问题是：为什么有些看似自然的递归程序会非常慢？

## 从切钢条问题开始

……

## 重复子问题：递归慢的真正原因

……

## 状态定义和状态转移

……

## 自顶向下：备忘录方法

……

## 自底向上：按顺序填表

……

## 这两种写法本质上有什么区别？

……

## 本节最容易混淆的地方

……
```

---

### 7.2 局部结构化块

主笔记不是完全不用结构，而是“主线自然，关键处结构化”。

例如公式可以这样写：

```markdown
状态定义：

\[
r[j] = 长度为 j 的钢条能获得的最大收益
\]

状态转移：

\[
r[j] = \max_{1 \le i \le j}(p[i] + r[j-i])
\]

这个式子的意思是：枚举第一段切成长度 \(i\)，这一段收益是 \(p[i]\)，剩余长度 \(j-i\) 的最优收益是 \(r[j-i]\)，在所有切法中取最大值。
```

算法可以这样写：

```markdown
这个伪代码可以分成三段看：

1. 初始化：准备数组 r，表示每个长度的最优收益；
2. 外层循环：按长度从小到大计算；
3. 内层循环：枚举第一刀切在哪里，取最大收益。

这样写的关键是：当我们计算 r[j] 时，所有更小的 r[j-i] 都已经算好了。
```

图示可以这样写：

```markdown
这张图的重点不是每个箭头的具体位置，而是说明：一个大问题的递归树中，会反复出现相同规模的子问题。图中重复出现的节点，就是动态规划能够节省计算的地方。
```

## 8. 覆盖报告设计

覆盖报告是产品差异化核心。

每个 block 必须有一个去向：

| 状态 | 含义 |
|---|---|
| covered | 已进入主笔记并解释 |
| merged | 合并到其他知识点中 |
| appendix | 放入附录或复习包 |
| ignored | 确认是噪声，如页码、装饰线、重复标题 |
| uncertain | 不确定，需要人工确认 |
| missing | 疑似遗漏 |

示例：

```markdown
# 覆盖报告

## 总览

- 有效 blocks：132
- 已覆盖：118
- 合并处理：8
- 放入附录：3
- 不确定：2
- 疑似遗漏：1

## 逐页覆盖情况

| 页码 | Block | 类型 | 状态 | 去向 | 说明 |
|---|---|---|---|---|---|
| p.1 | b001 | title | covered | overview | 合并进入总览 |
| p.3 | b004 | formula | covered | lecture_notes#状态转移 | 已解释符号含义 |
| p.5 | b002 | figure | covered | figures#fig-2 | 已解释图意 |
| p.7 | b006 | annotation | uncertain | - | 小字注释含义不明确 |
```

## 13. 处理流程伪代码

```python
def build_course_notes(file):
    project = create_project(file)

    # 1. 文档解析
    pages = parse_document(file)
    save_pages(project.id, pages)

    # 2. 构建 PageIR
    page_irs = build_page_ir(pages)

    # 3. 全局扫描
    global_outline = global_scan(page_irs)

    # 4. 逐页抽取知识单元
    page_units = []
    for page_ir in page_irs:
        units = extract_units_from_page(
            page_ir=page_ir,
            global_outline=global_outline
        )
        page_units.extend(units)

    # 5. 跨页合并
    merged_units = merge_units_across_pages(page_units)

    # 6. 建立知识关系
    relations = build_relations(merged_units)

    # 7. Note Director 规划写作策略
    note_plan = direct_notes(
        units=merged_units,
        relations=relations,
        global_outline=global_outline
    )

    # 8. 生成讲义式主笔记
    note_sections = []
    for section_plan in note_plan.sections:
        section = generate_lecture_section(
            section_plan=section_plan,
            course_ir_context=merged_units
        )
        note_sections.append(section)

    lecture_note = integrate_sections(note_sections)

    # 9. 生成结构化复习包
    review_pack = generate_review_pack(merged_units, relations)

    # 10. 覆盖率校验
    coverage = verify_coverage(
        blocks=collect_blocks(page_irs),
        units=merged_units,
        notes=[lecture_note, review_pack]
    )

    # 11. 局部修补
    if coverage.has_missing_items:
        patches = generate_missing_patches(coverage.missing_items)
        lecture_note = apply_patches(lecture_note, patches)

    # 12. 导出
    export_all(
        project_id=project.id,
        lecture_note=lecture_note,
        review_pack=review_pack,
        coverage=coverage
    )

    return lecture_note, review_pack, coverage
```

## 17. 最容易失败的地方

### 17.1 只做成 ChatPDF

如果产品主要界面就是一个聊天框，那它会很快变成普通文档问答工具。

你真正要做的是：

```text
课件 → 学习材料
```

而不是：

```text
课件 → 聊天
```

---

### 17.2 只保存最终笔记

只保存最终笔记会导致无法查漏、无法溯源、无法局部重生成。

必须保存：

```text
blocks
knowledge_units
relations
note_plan
note_chunks
coverage_items
```

---

### 17.3 主笔记太模板化

固定模板适合后台校验，不适合直接展示。

必须加入 Note Director，让最终输出变成自然讲义式。

---

### 17.4 过早追求全自动完美

课件图表很复杂，第一版不应该假装全部自动正确。

应该允许：

```text
uncertain
needs_review
manual_fix
regenerate_section
```

---

### 17.5 过早做复杂知识图谱可视化

知识图谱很有用，但第一版不要沉迷可视化。

先让图谱服务于：

```text
学习顺序规划
前置知识提示
混淆概念对比
问答检索
```

## 18. 项目的真正差异化

普通工具卖的是：

```text
AI 帮你总结 PDF
```

你的项目应该卖的是：

```text
AI 帮你把老师课件重构成可学习、可复习、可检查的知识系统
```

真正差异化是四个东西：

### 18.1 Source Mapping

每个知识点都能回到原课件页码和区域。

### 18.2 Coverage Ledger

每个原始 block 都有去向，知道是否遗漏。

### 18.3 CourseIR

课件被转成结构化知识，而不是一篇普通文本。

### 18.4 Note Director

最终笔记不是死板模板，而是自然讲义式表达。

## 19. 推荐的第一版目标

> 上传一份 30-80 页 CS/算法/OS/机器学习课件 PDF，系统生成一份自然讲义式 Markdown 笔记，并附带概念表、公式表、算法表、来源索引和覆盖报告。

## 20. 最终总结

这个项目应该这样构建：

> 先把 PDF/PPT 解析成带页码、bbox、图表和公式的结构化 blocks；再用大模型逐页抽取知识单元，跨页合并成 CourseIR；然后建立知识之间的前置、推导、对比和应用关系；接着由 Note Director 决定讲解策略，把结构化知识写成自然讲义式主笔记；最后用 coverage ledger 检查每个原始 block 是否被覆盖，并让用户在三栏界面里对照原课件、笔记和覆盖报告进行学习与修正。

最小实现链路：

```text
PDF
→ PageIR / Blocks
→ KnowledgeUnits
→ CourseIR
→ Note Director
→ Lecture Notes
→ Review Pack
→ Coverage Report
```

最重要的一句话：

> **后台要像数据库一样严谨，前台要像好助教一样会讲。**
